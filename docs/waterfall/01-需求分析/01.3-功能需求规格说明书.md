## 2. 模块一：仪表盘与指挥界面

### 2.1 总体要求

- **技术**: NiceGUI，暗色主题默认
- **布局**: 单页仪表盘，响应式（最小 1280×720）
- **刷新**: 实时数据推送（WebSocket 或轮询，≤3 秒延迟）

### 2.2 功能需求列表

#### FR-DASH-001: 项目标题区

| 属性 | 描述 |
|------|------|
| 需求描述 | 仪表盘顶部显示项目名称、版本号、当前状态 |
| 输入 | `get_project()["name"]`、`get_project()["status"]` |
| 输出 | 标题栏 UI 组件 |
| 验收标准 | 1. 显示 "Solo-Ops-Platform V1.0"  <br>2. 显示当前状态标签（进行中/已完成/暂停）<br>3. 状态标签颜色：进行中=蓝色，已完成=绿色，暂停=黄色 |

#### FR-DASH-002: KPI 指标卡片

| 属性 | 描述 |
|------|------|
| 需求描述 | 4 个 KPI 卡片横向排列，显示关键运营指标 |
| 指标 | 1. 今日任务完成数 / 计划数 <br>2. 运行中 Agent 数 <br>3. 本月 Token 消耗 / 预算 <br>4. 下次周期性回顾倒计时 |
| 输入 | `get_today_tasks()`、`get_running_agents()`、`get_monthly_cost()`、`get_next_review()` |
| 输出 | 4 个卡片 UI 组件 |
| 验收标准 | 1. 4 列等宽排列 <br>2. 大数字 + 小标签格式 <br>3. Token 消耗超过 80% 时卡片边框变橙色 <br>4. 超过 100% 时变红色 |

#### FR-DASH-003: Agent 团队网格

| 属性 | 描述 |

| 验收标准 | 1. 12 个卡片，2 行 × 6 列 <br>2. 每个卡片：彩色圆形头像（short_id）+ 名称 + 角色 + 状态灯 + 进度条（如有）+ Provider pill + 累计成本 <br>3. 状态灯颜色：idle=灰色，running/monitoring=蓝色发光，done=绿色，error=红色 <br>4. 点击卡片展开 Agent 详情面板 |

#### FR-DASH-004: 实时日志面板

| 属性 | 描述 |
|------|------|
| 需求描述 | 深色终端风格日志面板，显示最近 50 条系统日志 |
| 输入 | `get_recent_logs(limit=50)` |
| 输出 | 日志面板 UI 组件 |
| 验收标准 | 1. 深色背景 `#0F172A`，圆角 12px <br>2. 固定高度 200px，可滚动 <br>3. 格式：`[HH:MM] Agent名: 消息` <br>4. 时间戳灰色，Agent 名蓝色粗体，消息白色 <br>5. 新日志自动滚动到底部 |

#### FR-DASH-005: CEO 指挥面板

| 属性 | 描述 |
|------|------|
| 需求描述 | 右侧固定面板，创始人与主 Agent 交互的入口 |

#### FR-DASH-006: 任务执行结果区

## 3. 模块二：主 Agent 调度系统

#### FR-MAIN-001: 意图解析

| 属性 | 描述 |
|------|------|
| 需求描述 | 解析创始人自然语言指令，识别任务类型和意图 |
| 输入 | 创始人原始文本指令 |
| 输出 | 意图分类结果：`{"intent": "dev|mkt|ops|str|query", "confidence": 0.0-1.0, "keywords": [...], "suggested_sop": "SOP-ID"}` |
| 验收标准 | 1. 支持中文自然语言输入 <br>2. 置信度 >0.7 时自动选择 SOP，≤0.7 时提示创始人确认 <br>3. 无法识别时提供选项列表 <br>4. 解析耗时 <2 秒（本地 3B 模型） |

#### FR-MAIN-002: SOP 选择

| 属性 | 描述 |
|------|------|
| 需求描述 | 根据意图解析结果，选择对应的 SOP 模板 |
| 输入 | 意图解析结果 |
| 输出 | SOP 模板实例（包含阶段定义、Agent 团队配置、决策点） |
| 验收标准 | 1. 12 个 SOP 全部可映射 <br>2. 模糊意图时，主 Agent 提供 2-3 个候选 SOP 供创始人选择 <br>3. SOP 选择后锁定，执行中不可切换（除非创始人明确指令） |

#### FR-MAIN-003: Agent 团队组建

| 属性 | 描述 |
|------|------|
| 需求描述 | 根据 SOP 阶段定义，自动组建所需 Agent 团队 |
| 输入 | SOP 模板实例、当前可用 Agent 列表 |
| 输出 | Agent 团队配置：`{"stage_1": ["agent_id_1", "agent_id_2"], "stage_2": [...]}` |
| 验收标准 | 1. 每个阶段所需 Agent 自动匹配 <br>2. Agent 忙时自动排队或提示冲突 <br>3. 支持并行阶段同时启动多个 Agent <br>4. 串行阶段按顺序启动 |

#### FR-MAIN-004: 优先级抢占与恢复

| 属性 | 描述 |
|------|------|
| 需求描述 | 高优先级任务可中断低优先级任务，被中断任务状态保存稍后恢复 |
| 输入 | 新任务优先级、当前运行任务列表 |
| 输出 | 抢占决策：`{"action": "preempt|queue|reject", "target_task": "task_id", "saved_state": {...}}` |
| 验收标准 | 1. P0 可抢占任何非 P0 任务 <br>2. P1 可抢占 P2/P3/P4 <br>3. 同优先级不抢占，按时间排队 <br>4. 被抢占任务状态自动保存到 SQLite <br>5. 被抢占任务恢复时从断点继续 |

#### FR-MAIN-005: 冲突协调

| 属性 | 描述 |
|------|------|
| 需求描述 | 当多个 Agent 或任务产生冲突时，**先启动 Agent 间讨论协商，协商不成再升级创始人** |
| 输入 | 冲突报告（资源冲突 / 数据冲突 / 优先级冲突 / 方案分歧） |
| 输出 | 协商结果或升级请求 |
| 讨论规则 | 1. 冲突双方 Agent 自动进入讨论室，3-5 轮内尝试达成一致 <br>2. 讨论过程实时记录到审计日志 <br>3. 简单冲突（资源/优先级）由主 Agent 裁决 <br>4. 复杂冲突（方案分歧/数据冲突）由审计 Agent 裁决 <br>5. 出现"致命"分歧时自动弹窗升级创始人 <br>6. 其他分歧事后知情，创始人可随时查看讨论记录 |
| 验收标准 | 1. 资源冲突：自动排队，创始人可见 <br>2. 数据冲突：先讨论，审计 Agent 裁决，记录原因 <br>3. 优先级冲突：按 P0>P1>P2>P3>P4 自动解决 <br>4. 方案分歧：Agent 间讨论 3-5 轮，未达成一致则升级创始人 <br>5. 无法自动协调时，弹窗提示创始人决策 <br>6. 所有讨论记录可查看、可审计 |

#### FR-MAIN-006: 能量感知调度

| 属性 | 描述 |
|------|------|
| 需求描述 | 根据创始人能量状态，自动调整任务建议 |
| 输入 | 创始人当前能量水平（0-100%） |
| 输出 | 任务建议调整：`{"suggestion": "continue|switch|rest", "recommended_tasks": [...]}` |
| 验收标准 | 1. 能量 ≥70%：推荐继续当前复杂任务 <br>2. 能量 30-70%：推荐中等难度任务 <br>3. 能量 <30%：P3/P4 自动暂停，P2 延后，仅保留 P0/P1 <br>4. 系统主动建议休息 |

## 4. 模块三：Agent 执行引擎

#### FR-AGENT-001: 阶段生命周期管理

| 属性 | 描述 |
|------|------|
| 需求描述 | 管理 SOP 中每个阶段的启动、执行、完成、失败、重试 |
| 输入 | SOP 阶段定义、Agent 输出 |
| 输出 | 阶段状态更新：`{"stage_id": "", "status": "pending|running|completed|failed", "output": {...}}` |
| 验收标准 | 1. 每个阶段有独立状态 <br>2. 阶段间依赖自动管理（并行/串行）<br>3. 阶段失败时自动重试 1 次，仍失败则升级 <br>4. 阶段输出自动传递给下一阶段 |

#### FR-AGENT-002: 工具调用框架

| 属性 | 描述 |
|------|------|
| 需求描述 | Agent 可调用的工具注册、发现、执行、审计 |
| 工具类型 | 文件操作、网络请求、数据库查询、代码执行、文档生成 |
| 验收标准 | 1. 每个工具注册时定义：名称、描述、参数 schema、权限级别 <br>2. Agent 调用工具前需权限校验 <br>3. 每次调用记录：时间、Agent、工具、参数、结果、耗时、Token 消耗 <br>4. 工具调用结果返回给 Agent 继续推理 |

#### FR-AGENT-003: 结果输出与格式化

| 属性 | 描述 |
|------|------|
| 需求描述 | Agent 执行结果的标准化输出和格式化 |
| 输出格式 | Markdown（默认）、JSON（结构化数据）、文件（.docx/.xlsx/.pptx/.pdf） |
| 验收标准 | 1. 文本结果自动格式化 Markdown <br>2. 结构化数据输出 JSON Schema 校验 <br>3. 文档生成调用对应工具（python-docx / openpyxl / python-pptx / reportlab）<br>4. 输出文件保存到 `output/` 目录，命名规则：`{task_id}_{stage_id}_{timestamp}.{ext}` |

#### FR-AGENT-004: 状态实时上报

| 属性 | 描述 |
|------|------|
| 需求描述 | Agent 执行过程中实时上报状态到仪表盘 |
| 上报内容 | 当前阶段、进度百分比、正在执行的操作、预计剩余时间 |
| 验收标准 | 1. 每 5 秒或每阶段完成时上报 <br>2. 上报数据写入 SQLite `agent_status` 表 <br>3. 仪表盘实时读取并更新 UI <br>4. 上报失败时重试 3 次，仍失败标记 Agent 为 error 状态 |

#### FR-AGENT-005: 人类在环决策点

| 属性 | 描述 |
|------|------|
| 需求描述 | SOP 中定义需要创始人确认的关键决策点 |
| 触发条件 | SOP 阶段定义中 `decision_point: true` |
| 输出 | 弹窗/通知：`{"stage_id": "", "question": "", "options": [...], "timeout": 3600}` |
| 验收标准 | 1. 决策点触发时，系统暂停该任务后续阶段 <br>2. 弹窗显示问题 + 选项（确认/驳回/修改）<br>3. 创始人 1 小时内未响应，系统发送提醒通知 <br>4. 创始人选择后，系统记录决策并继续执行 |

#### FR-AGENT-006: Agent 间通信与讨论

| 属性 | 描述 |
|------|------|
| 需求描述 | Agent 之间可发起讨论、协商、辩论，讨论过程透明可审计 |
| 讨论类型 | 1. **并行阶段内部讨论**：SOP 并行阶段中多个 Agent 协作前的方案对齐（如产品经理+架构师讨论需求）<br>2. **跨 Agent 主动协商**：Agent 执行中发现问题主动找相关 Agent 协商（如开发 Agent 发现架构问题找架构师）<br>3. **冲突多方辩论**：多方对同一问题有分歧时展开辩论（如营销 vs 审计 vs 内容 Agent 对内容合规的辩论） |
| 讨论规则 | 1. 每次讨论 3-5 轮自动结束，未达成一致则升级或裁决 <br>2. 讨论记录全部保存到审计日志 <br>3. 创始人可随时查看"Agent 讨论室"实时面板 <br>4. 每次 Agent 发言计入 Token 成本（本质为 LLM 调用）<br>5. 讨论中不得泄露其他 Agent 的专属记忆（需主 Agent 授权） |
| 通信协议 | Agent 间通过消息总线通信，消息格式：`{"from": "agent_id", "to": "agent_id|broadcast", "type": "discuss|negotiate|debate", "topic": "", "content": "", "round": 1, "max_rounds": 5}` |
| 验收标准 | 1. 并行阶段 Agent 自动进入讨论室，讨论完成后联合输出 <br>2. Agent 可主动发起跨 Agent 协商，被请求 Agent 自动响应 <br>3. 冲突时自动触发多方辩论，审计 Agent 可介入裁决 <br>4. 讨论过程在"Agent 讨论室"面板实时显示 <br>5. 所有讨论记录可审计、不可篡改 <br>6. 讨论 Token 成本计入任务总成本 |

## 12. 模块十一：项目管理

> **核心问题**：创始人如何发起、引入、配置、结束、备份、封存一个项目？

### 12.1 总体要求

- **项目定义**：S-O-P 中的"项目"是独立业务单元，拥有专属 Agent 团队、记忆空间、SOP 配置、数据隔离边界
- **项目类型**：产品开发（心域探险/心栖陪伴/心理歌曲）、内容系列、运营活动、战略计划
- **项目状态机**：`draft`（草稿）→ `active`（活跃）→ `paused`（暂停）→ `archiving`（归档中）→ `archived`（已归档）→ `sealed`（已封存）

### 12.2 功能需求列表

#### FR-PROJ-001: 项目创建/发起

| 属性 | 描述 |
|------|------|
| 需求描述 | 创始人创建新项目，触发 STR-001 立项 SOP |
| 输入 | 项目名称、项目类型、项目描述、预期目标 |
| 流程 | 1. 创始人填写项目基本信息 <br>2. 系统自动分配项目 ID（`proj_{timestamp}_{shortname}`）<br>3. 触发 STR-001 立项 SOP（可行性评估 → 资源规划 → 创始人确认）<br>4. 立项通过后项目状态变为 `active`，自动创建项目专属资源 |
| 自动创建资源 | 1. SQLite `projects` 表记录 <br>2. ChromaDB `proj_{id}_memory` Collection <br>3. 项目目录 `data/projects/{id}/` <br>4. 项目默认 Agent 团队配置（可从全局 12 个 Agent 中指定子集）<br>5. 项目默认 SOP 模板绑定 |
| 验收标准 | 1. 填写 4 个字段即可创建草稿 <br>2. 立项 SOP 自动触发，纵向流程可视 <br>3. 立项通过后自动初始化所有资源 <br>4. 项目卡片立即出现在仪表盘项目总览 |

#### FR-PROJ-002: 项目导入/引入

| 属性 | 描述 |
|------|------|
| 需求描述 | 将已有项目（外部代码库、文档集、历史任务）引入 S-O-P 管理 |
| 输入 | 项目路径/压缩包、项目类型、导入配置 |

#### FR-PROJ-003: 项目配置

| 属性 | 描述 |
|------|------|
| 需求描述 | 配置项目的专属 Agent 团队、SOP 模板、记忆策略、预算配额 |

#### FR-PROJ-004: 项目状态管理

| 属性 | 描述 |
|------|------|
| 需求描述 | 管理项目生命周期状态转换 |
| 状态机 | `draft` → `active` → `paused` → `archiving` → `archived` → `sealed` |
| 状态定义 | - `draft`：已创建，未通过立项 SOP <br>- `active`：正常运行，可触发 SOP <br>- `paused`：暂停，Agent 不处理本项目任务，但数据保留 <br>- `archiving`：归档中，执行归档流程（备份→清理→迁移）<br>- `archived`：已归档，数据在温层，可查看/恢复 <br>- `sealed`：已封存，数据在冷层，需特殊流程恢复 |
| 状态转换规则 | - `draft` → `active`：立项 SOP 通过 <br>- `active` ↔ `paused`：创始人一键切换，暂停时保存所有运行中任务状态 <br>- `active` → `archiving`：创始人手动触发，或闲置超 90 天自动提醒 <br>- `archiving` → `archived`：归档流程完成 <br>- `archived` → `sealed`：归档 180 天后自动迁移到冷层 <br>- `archived` → `active`：创始人手动恢复，从温层还原数据 <br>- `sealed` → `active`：创始人手动申请，审计 Agent 评估后恢复（需 1-2 分钟从冷层加载） |
| 验收标准 | 1. 状态转换按钮在仪表盘项目卡片上直接操作 <br>2. 非法状态转换给出明确提示 <br>3. 状态变更记录审计日志 <br>4. `paused` 时运行中任务自动保存断点 <br>5. `archiving` 时显示归档进度 |

#### FR-PROJ-005: 项目归档/结束

| 属性 | 描述 |
|------|------|
| 需求描述 | 项目完结时的归档流程，确保数据完整保存、资源清理 |
| 触发 | 创始人手动触发，或项目目标达成后审计 Agent 建议 |
| 归档流程 | 1. **数据备份**：项目全部数据打包（SQLite 记录 + ChromaDB Collection + 项目目录文件）<br>2. **知识提炼**：运行 OPS-006 强制提炼，生成最终知识晶体 <br>3. **教训提取**：审计 Agent 扫描项目全部审计日志，提取跨项目共享教训 <br>4. **资源清理**：释放项目占用的活跃资源（暂停 Agent 任务、清理临时文件）<br>5. **数据迁移**：项目数据从热层迁移到温层 <br>6. **归档报告**：生成项目总结报告（周期、消耗、产出、教训、知识晶体清单） |
| 验收标准 | 1. 归档流程 6 步全部执行，进度可视 <br>2. 归档包可独立导出（.zip 格式）<br>3. 归档报告自动生成 Markdown <br>4. 归档后项目卡片变为灰色，显示"已归档"标签 <br>5. 归档数据 100% 可恢复 |

#### FR-PROJ-006: 项目独立备份

| 属性 | 描述 |
|------|------|
| 需求描述 | 对单个项目进行独立备份，不依赖全局备份 |
| 输入 | 项目 ID、备份备注、是否包含历史审计日志 |
| 备份内容 | 1. 项目 SQLite 记录 <br>2. 项目 ChromaDB Collection <br>3. 项目目录文件 <br>4. 项目配置快照 <br>5. 可选：完整审计日志 |
| 备份格式 | `backup_proj_{id}_{timestamp}.zip`，内含 JSON 元数据 + 数据文件 |
| 验收标准 | 1. 一键备份，显示进度 <br>2. 备份文件保存到 `backups/projects/` <br>3. 备份列表显示时间、大小、备注 <br>4. 支持从项目备份独立恢复（不覆盖其他项目）<br>5. 备份文件可导出到外部存储 |

#### FR-PROJ-007: 项目封存

| 属性 | 描述 |
|------|------|
| 需求描述 | 长期不活跃项目从温层迁移到冷层，释放 SSD 空间 |

#### FR-PROJ-008: 项目恢复

| 属性 | 描述 |
|------|------|
| 需求描述 | 从归档或封存状态恢复项目到活跃状态 |
| 恢复来源 | 1. 温层归档（快速恢复，< 30 秒）<br>2. 冷层封存（慢速恢复，1-5 分钟）<br>3. 外部备份包（导入式恢复） |

#### FR-PROJ-009: 项目数据隔离

| 属性 | 描述 |
|------|------|
| 需求描述 | 不同项目的数据完全隔离，防止交叉污染 |
| 隔离层级 | 1. **SQLite 隔离**：项目数据独立表空间或前缀隔离 <br>2. **ChromaDB 隔离**：每个项目独立 Collection <br>3. **文件系统隔离**：每个项目独立目录 <br>4. **Agent 记忆隔离**：Agent 在不同项目中的记忆不互通 <br>5. **审计日志隔离**：项目操作记录带项目 ID 标签 |
| 隔离规则 | - 项目 A 的 Agent 记忆默认不可被项目 B 访问 <br>- 项目 A 的文档默认不可被项目 B 的 SOP 引用 <br>- 全局 Agent（审计 Agent、主 Agent）可跨项目读取元数据，但不可读取详细内容 <br>- 跨项目访问需 FR-PROJ-010 授权 |
| 验收标准 | 1. 项目 A 的向量检索不会返回项目 B 的数据 <br>2. 项目 A 的审计日志独立存储，可按项目筛选 <br>3. 项目 A 的文件路径不可被项目 B 的 Agent 访问 <br>4. 隔离违规访问触发审计告警 |

#### FR-PROJ-010: 项目间数据共享（经授权）

| 属性 | 描述 |
|------|------|
| 需求描述 | 经创始人明确授权后，允许跨项目共享特定数据 |
| 共享类型 | 1. **知识晶体共享**：项目 A 的知识晶体可被项目 B 引用 <br>2. **教训共享**：项目 A 的教训自动进入共享教训库（默认共享）<br>3. **Agent 记忆共享**：经授权后，Agent 在项目 A 的经验可用于项目 B <br>4. **文档共享**：经授权后，项目 A 的文档可被项目 B 的 SOP 引用 |
| 授权流程 | 1. 创始人选择"共享数据" → 选择源项目 → 选择数据类型 → 选择目标项目 <br>2. 系统显示共享影响范围（哪些 Agent 会访问、Token 成本影响）<br>3. 创始人确认后生成共享授权记录 <br>4. 授权可随时撤销，撤销后目标项目不再访问新数据 |
| 验收标准 | 1. 共享操作需创始人二次确认 <br>2. 共享记录可审计、可查看 <br>3. 撤销共享即时生效 <br>4. 共享数据在目标项目中标记来源 <br>5. 共享数据计入源项目的成本归属 |
