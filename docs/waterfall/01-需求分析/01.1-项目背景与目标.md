# 01.1 项目背景与目标

> **文档类型**: 瀑布式需求分析 —— 项目背景与目标  
> **版本**: V1.0  
> **状态**: 需求冻结  
> **日期**: 2026-06-06  
> **技术栈**: NiceGUI + Python + SQLite + ChromaDB + Ollama  
> **范围**: 全量 —— 12 个 SOP、12 个 Agent、完整数据层、完整前端

---

## 1. 项目背景

### 1.1 创始人画像

| 属性 | 描述 |
|------|------|
| 身份 | 心理咨询师，一人公司创始人，个体户资质 |
| 技术背景 | IT 小白，技术栈以低代码/无代码为主（Coze / Kimi / Streamlit / Flutter） |
| 性格特质 | 极度社恐，怕失败，追求秩序感，排斥 1V1 服务 |
| 产品倾向 | 倾向内容产品间接影响力 |
| 商业目标 | 2 年内月均 30,000 元被动收入 |
| 核心诉求 | 管理三件事：公司日常、三个项目、自己的状态 |

### 1.2 三个核心项目

| 项目代号 | 项目名称 | 类型 | 状态 |
|---------|---------|------|------|
| 心域探险 | 心理探索类内容/工具 | 内容产品 | 规划中 |
| 心栖陪伴 | 心理陪伴类服务 | 内容产品 | 规划中 |
| 心理歌曲 | 音乐疗愈类内容 | 内容产品 | 规划中 |

### 1.3 核心痛点（已验证）

1. **任务优先级混乱** —— 多项目并行，不知道现在该做什么
2. **能量/状态记录缺失** —— 没有机制记录自己的状态波动
3. **状态影响任务没有关联** —— 能量低时还在接高难度任务
4. **周期性回顾没有机制** —— 周回顾、月回顾靠记忆，经常漏掉
5. **AI 不能持续对话** —— 每次重新交代背景，上下文丢失
6. **版本号混乱** —— 项目迭代没有规范版本管理
7. **工具性能差 + 界面丑** —— Streamlit 体验不达标

### 1.4 历史教训

- **V0.1~V0.9**: 基于 Streamlit 的敏捷迭代，需求边做边改，代码债累积，架构反复推翻
- **V0.10.0**: 尝试 Streamlit 界面重构，但技术栈与长期目标（NiceGUI）不一致，决定废弃
- **决策**: 从 V1.0 开始，采用**瀑布式开发**——需求冻结、设计冻结、编码、测试、交付，不回头

## 2. 项目目标

### 2.1 总体目标

构建一个**一人公司 AI 指挥平台**（Solo-Ops-Platform，简称 S-O-P），作为创始人自用操作系统，实现：

> **"宁可花点时间也要先把它弄到我满意，然后才能踏踏实实做后续项目。"**

### 2.2 具体目标

| 目标编号 | 目标描述 | 验收标准 |
|---------|---------|---------|
| G1 | 任务有序管理 | 所有任务按 P0-P4 优先级自动排队，高优先级可抢占低优先级 |
| G2 | 状态可视可溯 | 纵向（单任务全流程）+ 横向（全公司业务进展）实时可视 |
| G3 | 层次化指挥 | 创始人只向主 Agent 发命令，主 Agent 自动组织专业 Agent 协同 |
| G4 | 驱动感 | 创始人有专属任务列表和日程，秘书 Agent 主动提醒 |
| G5 | 数据主权 | 所有数据本地存储，LLM 只能看到脱敏数据，原始数据不出本地 |
| G6 | 成本可控 | 月度 Token 预算 ¥300，每次操作消耗实时可见，80% 预警，100% 熔断 |
| G7 | 自我进化 | 同类型踩坑 ≥3 次自动提炼 SOP 补丁，Agent 能力可动态扩展 |
| G8 | 知识资产化 | 原始文档 → 知识图谱 → 知识晶体，自动提炼，不过期 |

### 2.3 非目标（明确排除）

| 编号 | 排除项 | 原因 |
|------|--------|------|
| NG1 | 不做 SaaS 产品 | 自用为主，不对外提供服务 |
| NG2 | 不做移动端 App | 桌面端优先，NiceGUI 桌面应用 |
| NG3 | 不做实时协作 | 一人公司，无多人协作需求 |
| NG4 | 不做云端部署 | 本地优先，数据不出门 |
| NG5 | 不做通用 AI 助手 | 专注公司运营，不做聊天机器人 |

## 3. 产品哲学

### 3.1 核心立场

> **AI 是杠杆，不是替身。**  
> **你是指挥官，不是旁观者。**  
> **宁可花点时间，也要把它弄到你满意。**

### 3.2 "顺手"四标准（创始人定义）

| 标准 | 定义 | 设计体现 |
|------|------|---------|
| **有序** | Agent 很清楚现在要干什么 | 12 个 SOP 模板，按工作性质执行，先讨论再干活 |
| **可视** | 全过程可视可溯源 | 纵向（单任务全流程）+ 横向（全公司业务进展） |
| **有层次** | 你只向主 Agent 发布命令 | 主 Agent 自行组织各专业 Agent 协同 |
| **有驱动** | 你自己也有任务列表和日程 | 秘书 Agent 记录私人日程 + 项目依赖（带时间窗口和提醒） |

### 3.3 设计原则

| 原则 | 实践 |
|------|------|
| **本地优先** | 80% 场景本地 3B 模型（0 Token），15% 本地处理，5% 送云端大模型 |
| **默认透明** | 所有操作可审计、可查看、可回溯 |
| **人类指挥** | Agent 执行，人类决策，不越权 |
| **数据主权** | 你的数据属于你，随时可导出、可删除 |
| **渐进信任** | 能力逐步开放，权限由你授予 |
| **拒绝黑箱** | 思考过程可见，决策逻辑可解释 |
| **成本可控** | 月度 ¥300 预算，每次操作消耗实时可见 |
| **顺手第一** | 宁可花点时间也要弄到你满意 |

## 4. 约束条件

### 4.1 技术约束

| 约束 | 说明 |
|------|------|
| 前端框架 | NiceGUI（Python 原生，Vue 级美观，暗色主题默认） |
| 后端语言 | Python 3.10+ |
| 结构化数据库 | SQLite |
| 向量数据库 | ChromaDB |
| 本地模型 | Ollama（qwen2.5:3b + nomic-embed-text） |
| 云端模型 | Kimi / Coze / DeepSeek（可选，5% 场景） |
| 部署方式 | 本地桌面应用，数据不出电脑 |

### 4.2 成本约束

| 约束 | 说明 |
|------|------|
| 月度 Token 预算 | ¥300 |
| 预警阈值 | 80%（¥240） |
| 熔断阈值 | 100%（¥300），暂停非 P0 任务 |
| 本地模型 | 0 Token 成本 |
| 优化策略 | 知识晶体替代全文、共享上下文、本地模板填充 |

### 4.3 硬件约束

| 层级 | 容量 | 用途 | 自动迁移 |
|------|------|------|---------|
| 热层 | 50GB 本地 SSD | 活跃晶体、当前项目、本地模型 | — |
| 温层 | 256GB 外接 SSD | 3-12 月文档、备份包 | 30 天 → 90 天 |
| 冷层 | 1TB 机械硬盘 | 1 年以上归档、原始文档 | 90 天 → 180 天 |

### 4.4 时间约束

- 不设定硬截止日期
- 遵循"宁可花点时间也要弄到我满意"原则
- 按瀑布阶段推进，每阶段验收通过后再进入下一阶段

## 5. 决策权分层

| 场景 | 决策权 | 处理方式 |
|------|--------|---------|
| 业务方向（做什么 / 不做什么 / 优先级） | **创始人** | 创始人拍板，审计 Agent 记录 |
| 技术方案（用什么技术 / 架构设计） | **审计 Agent** | 审计 Agent 裁决，创始人可质疑但需技术理由 |
| 质量标准（致命 / 警告 / 建议判定） | **审计 Agent** | 依据预定义标准 |
| 发布时间（什么时候上线） | **创始人** | 创始人决定，审计 Agent 评估风险后建议 |
| 资源分配（Agent 时间 / 优先级调整） | **主 Agent** | 根据规则自动调度，创始人可见可干预 |
| SOP 补丁生效（技术类） | **审计 Agent** | 自动生效，创始人事后知情 |
| 争议处理 | **讨论决定** | 业务问题创始人大，技术问题 Agent 大 |

## 6. 风险与假设

### 6.1 关键假设

| 编号 | 假设 | 验证方式 |
|------|------|---------|
| A1 | NiceGUI 能满足所有 UI 需求 | 原型验证阶段确认 |
| A2 | Ollama qwen2.5:3b 能覆盖 80% 本地场景 | 基准测试 |
| A3 | 创始人能学会 NiceGUI 基础操作 | 培训文档 + 手把手指导 |
| A4 | SQLite + ChromaDB 能支撑一人公司数据量 | 性能测试 |

### 6.2 已知风险

| 编号 | 风险 | 影响 | 缓解措施 |
|------|------|------|---------|
| R1 | NiceGUI 学习曲线高于 Streamlit | 开发周期延长 | 预留 Flutter 备选方案 |
| R2 | 本地 3B 模型能力不足以处理复杂任务 | 被迫增加云端调用，超预算 | 知识晶体 + 模板填充优化 |
| R3 | 12 个 SOP 全量实现工作量巨大 | 项目延期 | 瀑布式分阶段，每阶段可独立验收 |
| R4 | 创始人技术背景有限，维护困难 | 长期可用性下降 | 文档完备 + 审计 Agent 辅助 |
