## 实验 1：NiceGUI 原型验证

### 目标

验证 NiceGUI 能否在 1 周内做出让创始人满意的单页仪表盘原型。

### 任务清单

#### Day 1-2: 搭建基础框架

- [ ] 安装 NiceGUI 并创建最小可运行项目
- [ ] 配置暗色主题（全局 CSS 变量）
- [ ] 创建页面布局：顶部标题区 + 左侧主内容区 + 右侧 CEO 面板
- [ ] 验证响应式布局在 1280×720 下正常显示

**交付物**: `experiments/exp01_nicegui/day2_layout.py` —— 可运行的基础布局

#### Day 3-4: 实现核心组件

- [ ] **标题区**: 显示 "Solo-Ops-Platform V1.0-MVP" + 状态标签
- [ ] **KPI 卡片**: 4 个卡片（今日任务数 / 运行中 Agent 数 / 本月 Token / 下次回顾）
- [ ] **Agent 网格**: 3 个卡片（主 Agent / 审计 Agent / 开发 Agent），含头像、状态灯、进度条
- [ ] **日志面板**: 深色背景 `#0F172A`，模拟 10 条日志，自动滚动
- [ ] **CEO 面板**: 输入框 + 发送按钮 + 3 个快捷按钮（"进度如何？"/"成本正常吗？"/"查看日志"）

**交付物**: `experiments/exp01_nicegui/day4_dashboard.py` —— 完整仪表盘原型

#### Day 5: 创始人验收

- [ ] 创始人运行原型，5 分钟内给出反馈
- [ ] 记录所有修改意见（颜色、布局、字体、交互）
- [ ] 判断：满意 / 基本满意需微调 / 不满意需换方案

**验收标准**:
- **通过**: 创始人说"这就是我想要的"或"只有小地方要调"
- **有条件通过**: 创始人说"整体可以，但 X/Y/Z 要改" → 记录修改点，评估修改工作量
- **不通过**: 创始人说"这不是我想要的"或"感觉不对" → 启动 Flutter 备选方案评估

**交付物**: `experiments/exp01_nicegui/验收记录.md`

## 实验 2：Ollama 3B 基准测试

### 目标

验证 qwen2.5:3b 在 20 个典型任务上的实际覆盖率，确定成本模型是否成立。

### 测试任务集（20 个）

#### 类别 A: 意图解析（5 个任务）

| # | 输入 | 期望输出 |
|---|------|---------|
| A1 | "帮我分析心域探险的竞品" | `{"intent": "dev", "sub_intent": "竞品分析", "confidence": >0.7}` |
| A2 | "写一份小红书文案，主题是情绪管理" | `{"intent": "mkt", "sub_intent": "内容创作", "confidence": >0.7}` |
| A3 | "修复昨天发现的登录 Bug" | `{"intent": "dev", "sub_intent": "bug修复", "confidence": >0.7}` |
| A4 | "看看这个月花了多少钱" | `{"intent": "ops", "sub_intent": "财务查看", "confidence": >0.7}` |
| A5 | "下周要发什么内容" | `{"intent": "mkt", "sub_intent": "内容规划", "confidence": >0.7}` |

#### 类别 B: SOP 选择（5 个任务）

| # | 输入 | 期望输出 |
|---|------|---------|
| B1 | "我想给心域探险加一个情绪日记功能" | 匹配 DEV-001（新功能开发） |
| B2 | "登录页面点不动，报错 500" | 匹配 DEV-002（Bug 修复） |
| B3 | "这周做了什么，下周计划是什么" | 匹配 STR-002（周期性回顾） |
| B4 | "帮我写个 Python 函数，读取 CSV 文件" | 匹配 DEV-001（新功能开发） |
| B5 | "系统好像有点慢，看看哪里有问题" | 匹配 DEV-002（Bug 修复）或 DEV-001 |

#### 类别 C: 代码生成（3 个任务）

| # | 输入 | 期望输出 |
|---|------|---------|
| C1 | "写一个 Python 函数，用 SQLite 查询最近 10 条任务记录" | 可运行的 Python 代码，含错误处理 |
| C2 | "写一个 NiceGUI 组件，显示一个带进度条的卡片" | 可运行的 NiceGUI 代码 |
| C3 | "写一个正则表达式，匹配手机号 138xxxx8888 格式" | 正确的正则表达式 |

#### 类别 D: 内容创作（4 个任务）

| # | 输入 | 期望输出 |
|---|------|---------|
| D1 | "写一段 200 字的小红书文案，主题是'焦虑时如何自我安抚'" | 200 字左右，有标题，有 emoji，有 CTA |
| D2 | "给公众号文章写个标题，内容是关于心理咨询中的边界设定" | 3 个候选标题，有吸引力 |
| D3 | "写一段抖音视频脚本，前 3 秒要有钩子，主题是'社恐如何社交'" | 15-30 秒脚本，有画面描述 |
| D4 | "把这段文字改写成更口语化：'认知行为疗法是一种结构化的心理治疗方法'" | 口语化版本，保留核心信息 |

#### 类别 E: 数据分析（3 个任务）

| # | 输入 | 期望输出 |
|---|------|---------|
| E1 | "分析这段用户反馈，提取关键问题和建议：[反馈文本]" | 结构化输出：问题列表 + 建议列表 |
| E2 | "比较这两个方案的优点和缺点：方案 A 用 SQLite，方案 B 用 JSON 文件" | 对比表格，含适用场景建议 |
| E3 | "总结这段技术文档的核心要点：[文档文本]" | 5 条以内要点，不遗漏关键信息 |

### 评分标准

每个任务按以下标准评分：

| 分数 | 标准 | 说明 |
|------|------|------|
| 2 分 | 完全可用 | 输出直接可用，无需修改 |
| 1 分 | 基本可用 | 输出有 80% 正确，需少量修改 |
| 0 分 | 不可用 | 输出错误、跑不通、或完全不符合要求 |

**通过标准**: 总分 ≥ 24 分（即平均 1.2 分/任务，约 60% 任务基本可用）

### 测试流程

```
Day 1: 准备测试环境
  ├── 确认 Ollama 已安装，qwen2.5:3b 已下载
  ├── 准备 20 个测试用例的输入文本
  └── 准备评分模板（Excel/CSV）

Day 2-3: 执行测试
  ├── 对每个任务，调用 Ollama 3B 生成输出
  ├── 记录生成时间、Token 数、内存占用
  └── 保存所有输出到文件

Day 4: 评分与分析
  ├── 创始人（或 Kimi）对每个输出评分
  ├── 统计总分、各类别得分、平均生成时间
  └── 分析失败案例：为什么失败？是模型能力问题还是提示词问题？

Day 5: 结论与调整
  ├── 如果总分 ≥ 24: 通过，按实际得分调整成本模型
  ├── 如果总分 16-23: 有条件通过，优化提示词后复测关键失败案例
  └── 如果总分 < 16: 不通过，评估升级模型（7B）或增加云端预算
```

## 实验 3：内存压力测试

### 目标

验证 3 个常驻 Agent + Ollama 3B + ChromaDB + NiceGUI 在 4GB 内存笔记本上能否流畅运行 1 小时。

### 测试场景

模拟一个典型工作负载：

```
T+0:00  启动 S-O-P，加载 3 个常驻 Agent，Ollama 3B 加载到内存
T+0:05  创始人输入："帮我写一个 SQLite 查询函数"
T+0:10  主 Agent 解析意图 → 匹配 DEV-001
T+0:15  主 Agent 拉起开发 Agent
T+0:20  开发 Agent 调用 Ollama 3B 生成代码
T+0:45  开发 Agent 输出代码，保存到 output/
T+0:50  审计 Agent 检查代码质量
T+1:00  创始人输入："把刚才的代码改成按时间排序"
T+1:05  主 Agent 解析为修改指令
T+1:10  开发 Agent 重新生成代码
T+1:35  输出修改后的代码
T+1:40  创始人输入："总结一下今天做了什么"
T+1:45  主 Agent 匹配 STR-002（周期性回顾）
T+1:50  主 Agent 查询今日任务记录
T+2:00  生成今日回顾摘要
...（循环类似任务，持续 1 小时）
```

### 监控指标

| 指标 | 工具 | 目标值 | 可接受上限 |
|------|------|--------|-----------|
| 内存峰值 | `psutil` 或系统监控 | < 3.5GB | 3.8GB |
| 内存平均 | `psutil` | < 3.0GB | 3.5GB |
| CPU 峰值 | `psutil` | < 80% | 90% |
| UI 响应时间 | 手动测试 | < 2 秒 | 5 秒 |
| OOM 次数 | 系统日志 | 0 | 0 |
| 任务完成率 | 测试脚本统计 | 100% | 95% |

### 测试流程

```
Day 1: 准备测试脚本
  ├── 编写自动化测试脚本，模拟上述工作负载
  ├── 集成内存监控（每 5 秒记录一次）
  └── 准备 10 个不同的创始人输入，循环使用

Day 2: 执行测试
  ├── 关闭所有非必要程序（浏览器、微信等）
  ├── 启动测试脚本，持续运行 1 小时
  └── 期间创始人可正常使用电脑做其他事（模拟真实场景）

Day 3: 分析结果
  ├── 绘制内存使用曲线
  ├── 识别内存峰值时刻和原因
  ├── 检查是否有内存泄漏（内存是否持续增长）
  └── 得出结论
```

## 验证通过后的流程

### 全部通过

```
1. 更新 PRD 成本模型（基于实验 2 实际覆盖率）
2. 创始人确认更新后的 PRD
3. 冻结设计（进入 02-系统设计阶段）
4. 设计阶段交付物：
   ├── 02.1-系统架构设计.md
   ├── 02.2-数据库设计.md
   ├── 02.3-接口设计.md
   ├── 02.4-UI/UX 设计.md
   └── 02.5-安全设计.md
```

### 实验 1 不通过（NiceGUI 不满意）

```
1. 评估 Flutter 切换成本（预计增加 2-4 周学习+开发时间）
2. 创始人决策：坚持 NiceGUI（接受当前效果）/ 切 Flutter / 寻找其他方案
3. 根据决策调整技术栈约束
4. 重新验证（如需要）
```

### 实验 2 不通过（3B 覆盖率 < 60%）

```
1. 分析失败案例：是模型能力问题还是提示词问题？
2. 如果是提示词问题：优化提示词，复测关键失败案例
3. 如果是模型能力问题：
   ├── 选项 A: 升级本地模型到 7B（内存压力增加，需重新评估实验 3）
   ├── 选项 B: 增加云端预算（月度 ¥300 → ¥500，需创始人确认）
   └── 选项 C: 降低本地场景比例（60% → 40%，增加云端调用）
4. 创始人决策后调整成本模型
```

### 实验 3 不通过（内存 > 3.5GB 或 OOM）

```
1. 分析内存峰值原因：Ollama 占用？Agent 进程？ChromaDB？
2. 优化方案：
   ├── 选项 A: 减少常驻 Agent 到 2 个（主 Agent + 开发 Agent，审计 Agent 按需加载）
   ├── 选项 B: Ollama 用完即卸载（增加加载时间，减少常驻内存）
   └── 选项 C: 要求升级硬件（8GB 内存，需创始人确认）
3. 实施优化后重新测试
```
