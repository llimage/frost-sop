# Solo-Ops-Platform V0.8.0 需求文档

**版本：** V0.8.0 - 知识武装
**日期：** 2026-06-05
**定位：** 接入本地知识库，从"通用模型"升级为"懂你业务的专家"

---

## 一、版本目标

为平台引入本地知识库能力，使 Agent 能检索并引用私有文档。支持多种格式导入、向量化存储、语义+关键词混合检索，以及引用溯源。

**核心理念：** Agent 应该读过你的产品文档、竞品分析、行业报告——所有你积累的私有知识。

**不涉及：** 知识图谱、自动知识发现、外部知识库 API、多用户权限。留给后续版本。

---

## 二、知识库架构

```
/knowledge/
├── documents/               ← 原始文档（上限 1GB）
│   ├── product/             ← 产品文档
│   ├── competitors/         ← 竞品分析
│   ├── industry/            ← 行业报告
│   ├── internal/            ← 内部知识
│   ├── custom/              ← 用户自定义分类
│   └── .archive/            ← 旧版本归档（30天自动清理）
├── vector_store/            ← ChromaDB 持久化（或 SQLite 降级）
├── knowledge_index.json     ← 文档索引
├── import_log.json          ← 导入日志
└── seed/                    ← 种子文档
    └── 欢迎使用知识库.md
```

## 三、核心功能

### 3.1 文档导入

#### 支持格式

| 格式 | 解析方式 |
|------|----------|
| `.txt` | 直接读取，UTF-8 编码 |
| `.md` | 保留标题层级作为结构信息 |
| `.pdf` | pypdf（轻量级方案） |

#### 导入方式

**方式A：前端上传。** 侧栏知识库区域点击上传，选择文件并指定分类。

**方式B：文件 Skill 自动导入。** 利用 V0.4.0 文件 Skill，Agent 可将生成的文件自动归档。

#### 文档处理流程

```
原始文档
    │
    ▼
格式解析（txt/md/pypdf）
    │
    ▼
敏感信息检测（置信度评分，非硬阻断）
    │
    ▼
智能分块（按类型自动选择策略）
    │
    ▼
向量化（sentence-transformers）
    │
    ▼
存入向量库 + BM25 索引 + 更新 knowledge_index.json
```

---

### 3.2 智能分块策略

按文档特征自动选择分块模式，用户可手动覆盖：

| 条件 | 判定类型 | chunk_size | overlap |
|------|----------|-----------|---------|
| 文件名含 faq/问答/Q&A | 问答 | 300 | 30 |
| 首行以 "Q:" 或 "问：" 开头 | 问答 | 300 | 30 |
| .md 且含多级标题（≥3个#） | 标准 | 500 | 50 |
| 文件大小 > 200KB | 长文档 | 800 | 80 |
| 其余 | 标准 | 500 | 50 |

**中文优化：** 优先按段落和标题切分。当按字符切分时，寻找切分点的优先级：`\n\n` > `\n` > `。` > `！` > `？` > `，`。确保不截断语义。

### 3.3 向量化

**默认方案：paraphrase-multilingual-MiniLM-L12-v2**

- 多语言模型，中文语义检索效果好
- 384 维向量，本地运行，零 API 成本
- 首次使用时自动下载（约 120MB），使用镜像源 `HF_ENDPOINT=https://hf-mirror.com`
- 下载超时 120 秒，失败时提示用户检查网络并支持手动重试
- 后续离线使用

**缓存路径配置（运行时设置）：**

```python
# 在 init_knowledge_system() 中执行
os.environ["HF_HOME"] = str(Path(__file__).parent.parent / ".hf_cache")
os.environ["TRANSFORMERS_CACHE"] = str(Path(__file__).parent.parent / ".hf_cache")
Path(os.environ["HF_HOME"]).mkdir(parents=True, exist_ok=True)
```

缓存路径：`Solo-Ops-Platform/.hf_cache/`，需加入 `.gitignore`。

### 3.4 混合检索（Hybrid）

```
用户查询
    │
    ├── 语义检索（ChromaDB/SQLite 向量相似度）
    │        │
    ├── 关键词检索（BM25 + jieba 分词）
    │        │
    └── RRF 融合（Reciprocal Rank Fusion，k=60）
             │
             ▼
        排序后的 Top-K 结果
```

**检索接口参数：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `query` | 必填 | 查询文本 |
| `top_k` | 5 | 返回结果数 |
| `category` | None | 按分类过滤 |
| `semantic_weight` | 0.7 | 语义检索权重 |
| `keyword_weight` | 0.3 | 关键词检索权重 |

### 3.5 向量库后端（双模式）

**启动时自动检测：**

```python
def _detect_backend():
    # 检测 ChromaDB 持久化能力
    if _test_chroma_persistence():
        return "chromadb"
    else:
        logger.info("ChromaDB 不可用，降级为 SQLite + 内存向量方案")
        return "sqlite"
```

| 后端 | 条件 | 持久化 | 检索方式 |
|------|------|--------|----------|
| ChromaDB | 启动检测通过 | 磁盘持久化 | 内置向量检索 |
| SQLite + numpy | ChromaDB 不可用时自动降级 | SQLite 持久化 | 内存余弦相似度（暴力搜索，<10000 篇适用） |

**用户无感知切换。** 前端显示当前后端状态（ChromaDB 或降级模式）。

#### SQLite 降级方案实现要点

- **向量存储：** `vectors` 表，`vector` 字段使用 `numpy.tobytes()` 存储为 BLOB
- **检索时：** 启动时全量加载到内存字典 `{chunk_id: np.ndarray}`
- **相似度计算：** `np.dot(query_vec, all_vecs) / (norm(query_vec) * norm(all_vecs).reshape(-1,1))`
- **上限判断：** `chunks` 表 `COUNT > 10000` 时，前端提示用户"知识库规模较大，建议安装 ChromaDB 以获得更好性能"
- **新增依赖：** `numpy`（Python 3.13 标准环境已自带，无需额外添加）

### 3.6 BM25 索引一致性保障

```python
def _verify_index_consistency():
    chroma_count = collection.count()  # O(1)
    bm25_count = len(bm25_index.documents)

    if chroma_count != bm25_count:
        with file_lock(LOCK_PATH, timeout=30):
            # 二次检查（获取锁后可能已被其他进程重建）
            if collection.count() != len(bm25_index.documents):
                _rebuild_bm25_from_chroma(progress_callback=_log_progress)
```

**规则：** BM25 是 ChromaDB/SQLite 的派生索引。启动时自动校验，不一致时以向量库为准重建 BM25。重建加文件锁防并发。

### 3.7 引用溯源

每条检索结果附带来源：

```json

```

Agent 可在生成输出时标注引用来源文档。

## 四、安全体系

### 4.1 容量管理

| 限制项 | 阈值 | 处理方式 |
|--------|------|----------|
| 单文件大小 | 10MB | 超出拒绝导入 |
| documents/ 总容量 | 1GB | 超出时提示清理，24h 未响应自动清理 .archive/ |
| 文档类型 | .txt .md .pdf | 白名单外拒绝 |

### 4.2 敏感信息检测（置信度评分）

```python
SENSITIVE_PATTERNS = {
    "身份证号": (r'\d{17}[\dXx]', 0.9),
    "手机号":   (r'1[3-9]\d{9}', 0.3),
    "银行卡号": (r'\d{16,19}', 0.5),
}
```

| 置信度 | 处理 |
|--------|------|
| > 0.8（高） | 阻塞导入，提示用户人工确认 |
| 0.5-0.8（中） | 警告但允许继续 |
| < 0.5（低） | 静默通过 |

仅展示检测到的模式类型和数量，不展示具体匹配内容（避免二次泄露）。

### 4.3 并发控制

导入操作加文件锁（`filelock`），锁超时 30 秒。两个 Agent 同时导入时排队执行。

### 4.4 路径安全

向量存储路径、文档路径均纳入 `safe_path()` 校验，确保在 `PROJECT_ROOT` 内。

## 九、CEO 适配

### 9.1 知识检索步骤

CEO 在 Plan 阶段，记忆检索之后增加知识检索：

```
[Memory] 正在检索相关历史经验...
[Memory] 找到 2 条相关经验
[Knowledge] 正在检索相关知识...
[Knowledge] 找到 3 篇相关文档：竞品A分析报告.md、产品功能清单.md...
```

**优化：** 如果知识库为空，跳过知识检索步骤，日志显示 `[Knowledge] 知识库为空，跳过检索`。

### 9.2 知识注入

检索结果按分类整理，注入 CEO 系统提示词，放在记忆注入之后。

### 9.3 长度限制

单次知识注入上限：**2000 字符**。超出时按 RRF 得分从高到低拼接片段，每个片段截取前 300 字符，直至达到上限。

### 9.4 引用指令

CEO 在 task_description 中增加："如果引用了知识库内容，请在输出中标注来源文档名称"。

### 9.5 知识库与记忆系统冲突处理规则

- **优先级：** 记忆系统（历史经验）> 知识库（静态文档）
- **原因：** 记忆是已验证有效的任务经验总结，知识库文档可能过时
- **冲突检测：** 当记忆和知识库对同一问题给出相反建议时，CEO 在输出中标注：
  > "注意：知识库 [文档名] 中的建议与历史经验不符，建议优先参考历史经验。"
- **注入顺序：** 先注入记忆，再注入知识库。知识库注入时附加提示："以下内容供参考，可能与最新实践有差异。"

## 十四、验收标准

### 文档导入

1. 上传 `.txt` 和 `.md` 文件后自动分块和向量化
2. 上传 `.pdf` 文件后 pypdf 正确提取文本
3. 切分后每个 chunk 的首尾 50 字符不包含断句字符，分割点位于段落或句子边界

### 检索能力

4. 混合检索返回语义+关键词融合结果，RRF 排序
5. 检索结果附带来源信息（文档名、分类）
6. 按分类过滤检索正确

### CEO 集成

7. CEO 日志显示 `[Knowledge]` 检索步骤（知识库为空时显示跳过）
8. 任务输出可标注知识来源
9. 知识注入按 RRF 得分排序拼接，累计字符数 ≤ 2000

### 安全与稳定性

10. 删除文档同步清理向量和 BM25 索引
11. 敏感信息检测：高置信度阻塞，中置信度警告，低置信度通过
12. BM25 索引启动时自动校验一致性

### 向后兼容

13. V0.7.0 核心功能正常：驾驶舱加载、新建任务（CEO+固定流水线）、历史任务列表/详情/下载、产品线管理、Skill 工具全部可调用
14. 记忆系统正常，不受影响
