## 四、已完成的文件改动

### 4.1 `skills/llm.py` — 参数锁

**改动位置**：`_call_online_llm` 函数，第283行附近

**新增逻辑**：
```python
# 基于任务类型的 temperature 映射
_llm_profile = context.get("_llm_profile", "")
if _llm_profile == "execute":
    default_temp = 0.1      # 执行类：高确定性
elif _llm_profile == "create":
    default_temp = 0.5      # 创意类：适度发散
elif _llm_profile == "review":
    default_temp = 0.2      # 审查类：高一致性
else:
    default_temp = 0.3      # 默认：比原0.7更保守

if "_temperature" not in context:
    temperature = default_temp
```

**向后兼容**：如果调用方显式传了 `_temperature`，仍尊重用户选择。

### 4.2 `sops/templates/OPS-007.yaml` — 输出模板

**每个阶段新增**：
- `llm_config`：指定 `profile: execute`，自动映射到 temperature=0.1
- `output_template`：定义该阶段必须输出的结构（Markdown/JSON格式）
- `constraint`：硬性约束（如"实施难度只能是简单/中等/复杂"）

**全局新增**：
```yaml
llm_config:
  profile: execute
  top_p: 0.8
  max_tokens_per_phase: 2000
```

### 4.3 `config/AGENTS.md` — 约束宪法

**13条规则**，分三层：
- **输出格式约束**（5条）：禁止引言、JSON必须可解析、量化数据必须真实等
- **推理过程约束**（4条）：不确定时说"不确定"、优先量化数据、必须明确判断等
- **任务执行约束**（3条）：严格按SOP执行、符合output_template、遇错停止等

**加载规则**：每次任务执行前，编排器读取此文件注入system prompt。

## 五、诚实评估：哪些指标没改善？

| 指标 | 基线 | 改进后 | 评估 |
|------|------|--------|------|
| Jaccard相似度（Run1 vs Run3） | 33.33% | 33.33% | ⚠️ 未改善——方法2措辞仍有变化 |
| Token消耗 | 147 | 150 | ~持平 |
| 概念完全一致性 | 低 | **中** | 方法名称一致，但描述措辞仍有差异 |

**原因分析**：
- Jaccard是词汇级指标，对同义替换敏感（"微调与对齐" vs "微调与RLHF"是不同词汇但同义概念）
- 描述措辞的差异源于prompt要求"50字以内说明"——模型在压缩措辞时有不同选择
- **进一步的改善空间**：在output_template中强制使用**固定字段名**（如`method_name`、`principle`、`effect`），而非自由文本描述，可消除措辞差异

## 六、下一步建议（优先级排序）

### P0 — 本周必做（验证方向是否正确）

1. **用改进后的OPS-007执行一次真实狩猎任务**

   - 每次调用前，自动将AGENTS.md内容注入system prompt

### P1 — 下周可做（进一步提升）

3. **给output_template增加JSON Schema版本**
   - 当前output_template是文本描述，LLM仍有一定自由
   - 进阶：用Pydantic模型定义schema，强制字段名和类型
   - 预期效果：Jaccard相似度提升到70%+

   - 每个阶段执行后，自动检查输出是否符合output_template
   - 不符合时自动重试（最多2次）
   - 预期效果：格式合规率从~80%提升到~95%

### P2 — 下月考虑（深度优化）

5. **SOP模板批量改进**

6. **工具调用（web_fetcher）**
   - 让Agent调用真实工具获取数据，而非凭记忆回答
   - 预期效果：事实幻觉大幅降低
