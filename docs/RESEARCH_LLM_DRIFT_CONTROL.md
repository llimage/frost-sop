# 大模型"思维飘逸"可控性深度调研

**完成日期**: 2026-07-04
**来源**: 独立第三方深度调研
**核心结论**: 约束比能力更重要，飘逸无法消除但可以管理

---

## 核心发现

### 发现1: 思维飘逸的本质

**不是幻觉，是推理轨迹的系统性偏移**

- **漏斗效应**: 早期步骤决定最终结果，一旦进入错误路径，后期纠错可能性急剧下降
- **模型特异性漂移**: Llama陷入"惯性漂移"，Qwen表现为"混沌漂移"
- **通用故障原型**: 所有模型都(1)从问题直接跳到求解，(2)系统性回避自我反思

### 发现2: 约束悖论（最重要的发现）

**LangChain仅通过修改harness（不加模型、不微调），在Terminal Bench 2.0上从52.8%提升至66.5%（+13.7pp）**

三个关键变更：
1. **LocalContextMiddleware**: 在每一步重新注入关键上下文
2. **PreCompletionChecklistMiddleware**: 强制模型检查是否满足所有约束
3. **计算预算管理**: 限制推理步数（更多思考≠更好结果）

**反直觉结论**: 最大推理预算（xhigh）反而比基线更差。

## 对FROST-SOP的具体建议

### 第一优先级: 三层防御架构

```
Layer 3: FSM编排层（确定性）
- 状态转换规则（不允许LLM自行决定跳转）
- 前置条件验证（每步执行前检查合法性）
- 超时/迭代上限（防止死亡循环）

Layer 2: 结构化约束层（半确定性）
- Structured Outputs（每步输出Schema强制）
- Function Calling白名单+速率限制
- AGENTS.md/CLAUDE.md约束文件

Layer 1: LLM执行层（非确定性）
- 在Layer 3定义的状态边界内自由推理
- Temperature 0.2-0.3（低创造性，高一致性）
```

### 第二优先级: 失败日志驱动的持续约束积累

每发现一次Agent飘逸，就在约束文件中添加一条规则。

### 第三优先级: 混合架构而非纯LLM

高价值低容错场景必须用"LLM+规则引擎"混合架构。

---

## 不要做的事

❌ 不要追求"消除飘逸"——理论上不可能
❌ 不要给Agent无限推理预算——更多思考≠更好结果
❌ 不要让LLM自主管理记忆——引入额外不可控性
❌ 不要在SOP中堆砌规则——遵循率指数衰减
❌ 不要依赖单一模型——模型静默更新可导致行为突变

---

## 最核心的一句话

**在2026年，大模型可控性的真实水平是——飘逸无法消除但可以管理，管理的核心不是更强的模型，而是更紧的约束。**
