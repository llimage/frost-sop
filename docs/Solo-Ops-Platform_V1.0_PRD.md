## 1. 项目概述

### 1.1 项目背景

Solo-Ops-Platform（简称 S-O-P）是一个**一人公司 AI 指挥平台**，由心理咨询师廖亮作为创始人自用开发。

**历史教训**：
- V0.1~V0.9：基于 Streamlit 的敏捷迭代，需求边做边改，代码债累积，架构反复推翻
- V0.10.0：尝试 Streamlit 界面重构，但技术栈与长期目标（NiceGUI）不一致，决定废弃
- **V1.0 教训**：第三方审计指出"全量 69 条 FR 不可交付"，修正为 **MVP 优先 + 假设验证前置**

### 1.2 创始人画像

| 属性 | 描述 |
|------|------|
| 身份 | 心理咨询师，一人公司创始人，个体户资质 |
| 技术背景 | IT 小白，技术栈以低代码/无代码为主 |
| 性格特质 | 极度社恐，怕失败，追求秩序感，排斥 1V1 服务 |
| 产品倾向 | 倾向内容产品间接影响力 |
| 商业目标 | 2 年内月均 30,000 元被动收入 |
| 核心诉求 | 管理三件事：公司日常、三个项目、自己的状态 |

### 1.3 核心痛点（MVP 必须解决）

1. **任务优先级混乱** —— 多项目并行，不知道现在该做什么
2. **AI 不能持续对话** —— 每次重新交代背景，上下文丢失
3. **状态影响任务没有关联** —— 能量低时还在接高难度任务
4. **周期性回顾没有机制** —— 周回顾、月回顾靠记忆，经常漏掉
5. **工具性能差 + 界面丑** —— Streamlit 体验不达标

### 1.4 MVP 目标

| 目标编号 | 目标描述 | 验收标准 |
|---------|---------|---------|
| G1 | 任务有序管理 | 3 个 SOP 按阶段执行，进度实时可见 |
| G2 | 状态可视可溯 | 仪表盘显示 Agent 状态、任务进度、Token 消耗 |
| G3 | 层次化指挥 | 创始人只向主 Agent 发命令，主 Agent 拉起开发/审计 Agent 执行 |
| G4 | 驱动感 | 创始人有任务列表和日程，秘书功能由主 Agent 兼任 |
| G5 | 数据主权 | 所有数据本地存储，API Key 加密 |
| G6 | 成本可控 | 月度 Token 预算 ¥300，实时显示消耗 |

### 1.5 非目标（MVP 明确不做）

| 编号 | 排除项 | 原因 |
|------|--------|------|
| NG1 | 不做 SaaS 产品 | 自用为主 |
| NG2 | 不做移动端 App | 桌面端优先 |
| NG3 | 不做实时协作 | 一人公司 |
| NG4 | 不做云端部署 | 本地优先 |
| NG5 | 不做通用 AI 助手 | 专注公司运营 |
| **NG6** | **不做 12 个常驻 Agent** | **内存不够，V1.0 只做 3 个常驻** |
| **NG7** | **不做 12 个 SOP** | **V1.0 只做 3 个核心 SOP，其余 V1.1+ 迭代** |
| **NG8** | **不做知识晶体生命周期** | **V1.0 用文档摘要+标签替代** |
| **NG9** | **不做自动分层存储迁移** | **V1.0 手动归档即可** |
| **NG10** | **不做自我进化（自动提炼 SOP 补丁）** | **V1.0 手动记录教训** |
| **NG11** | **不做收入追踪系统** | **中期任务留存，V1.1+ 实现** |

## 2. 产品哲学与宣言

### 2.1 核心立场

> **AI 是杠杆，不是替身。**  
> **你是指挥官，不是旁观者。**  
> **宁可花点时间，也要把它弄到你满意。**

### 2.2 "顺手"四标准

| 标准 | 定义 | MVP 设计体现 |
|------|------|-------------|
| **有序** | Agent 很清楚现在要干什么 | 3 个 SOP 模板，按阶段执行 |
| **可视** | 全过程可视可溯源 | 仪表盘：Agent 网格 + 日志面板 + 任务结果 |
| **有层次** | 你只向主 Agent 发布命令 | 主 Agent 解析意图，拉起开发/审计 Agent |
| **有驱动** | 你自己也有任务列表和日程 | 今日任务列表 + 能量状态记录 |

### 2.3 修正后的设计原则

| 原则 | MVP 实践 |
|------|---------|
| **本地优先** | 意图解析、SOP 选择、简单任务用本地 3B；复杂任务送云端 |
| **默认透明** | 操作日志追加-only，Token 消耗实时显示 |
| **人类指挥** | **技术决策权归创始人**，审计 Agent 仅提供建议 |
| **数据主权** | 数据本地存储，随时可导出 SQLite 文件 |
| **成本可控** | 月度 ¥300 预算，超 80% 预警，超 100% 熔断 |
| **顺手第一** | 3 个月交付 MVP，能用起来再迭代 |

## 3. MVP 范围定义

### 3.1 功能范围：25 条 FR

#### 核心功能（15 条）—— 没有这些系统不能用

| 模块 | 数量 | 功能 |
|------|------|------|
| 仪表盘 | 6 | 标题区、KPI 卡片、Agent 网格（3 个）、日志面板、CEO 面板、任务结果区 |
| 主 Agent 调度 | 3 | 意图解析、SOP 选择、Agent 团队组建 |
| Agent 执行引擎 | 4 | 阶段生命周期、工具调用框架、结果输出格式化、状态实时上报 |
| 记忆系统 | 2 | Agent 专属记忆（简化）、向量语义检索 |

#### 支撑功能（10 条）—— 系统能跑起来需要这些

| 模块 | 数量 | 功能 |
|------|------|------|
| 审计日志 | 2 | 操作审计（追加-only）、成本审计 |
| 成本管理 | 2 | 月度预算配置、实时成本追踪 |
| 状态管理 | 2 | 能量状态记录（简化）、私人日程管理 |
| 数据存储 | 2 | SQLite 结构化数据、ChromaDB 向量数据 |
| 设置 | 2 | API 配置、本地模型配置 |

### 3.2 Agent 架构：3 个常驻 + 9 个角色模板

| 类型 | 数量 | 说明 |
|------|------|------|
| **常驻 Agent（内存中始终运行）** | 3 | 主 Agent（调度）、审计 Agent（监督）、开发 Agent（执行） |
| **角色模板（磁盘定义，需要时实例化）** | 9 | 产品经理、架构师、测试、营销、内容、运营、财务、行政、秘书 |
| **内存约束** | — | 同时运行 Agent ≤ 3 个，Ollama 3B 按需加载，总内存 ≤ 3GB |

### 3.3 SOP 范围：3 个核心 SOP

| SOP ID | 名称 | 为什么先保留 |
|--------|------|-------------|
| DEV-001 | 新功能开发 | 你在开发 S-O-P 本身，这是最常用的 |
| DEV-002 | Bug 修复 | 软件项目必备 |
| STR-002 | 周期性回顾 | 解决"周期性回顾没有机制"的核心痛点 |

**其他 9 个 SOP（MKT-001/002/003, OPS-001~006, STR-001/003）V1.1~V1.x 逐步添加。**

### 3.4 砍掉的功能（V1.1+ 再考虑）

| 原功能 | 砍掉原因 | 替代方案 |
|--------|---------|---------|
| 项目总览横向视图 | MVP 只需看当前任务 | V1.1 加 |
| Agent 讨论室面板 | 3 个常驻 Agent 直接通信即可 | V1.1 加 |
| 能量状态输入器（6 种情绪按钮） | 简化为滑动条+文本 | V1.1 加情绪按钮 |
| 今日日程时间线 | 私人日程管理够用 | V1.1 加时间线可视化 |
| 优先级抢占与恢复 | MVP 串行执行即可 | V1.1 加 |
| 冲突协调（Agent 间讨论） | 3 个 Agent 冲突概率低 | V1.1 加 |
| 能量感知调度 | 手动记录能量即可 | V1.1 加自动建议 |
| 人类在环决策点（复杂弹窗） | 简化为主 Agent 询问确认 | V1.1 加决策点框架 |
| 共享教训库 | 手动记录到 SQLite 即可 | V1.1 加自动归集 |
| 知识晶体生命周期 | 文档摘要+标签够用 | V1.1 加 6 阶段生命周期 |
| 上下文注入 | 手动检索记忆即可 | V1.1 加自动注入 |
| 质量审计 | 开发 Agent 自检 + 审计 Agent 抽查 | V1.1 加正式质量审计 |
| 审计报告生成 | 日志可查看即可 | V1.1 加报告生成 |
| 成本优化建议 | 手动控制预算即可 | V1.1 加自动建议 |
| 项目依赖日程 | 私人日程管理够用 | V1.1 加 |
| 周期性回顾自动触发 | 手动触发 STR-002 即可 | V1.1 加定时触发 |
| 工作/休息边界提醒 | V1.0 不实现 | V1.1 加每小时通知 |

## 4. 假设验证计划

### 4.1 为什么先验证再冻结

第三方审计指出：**"需求还没稳定，就宣布'需求冻结'，是自欺欺人。"**

修正方案：**进入设计阶段之前，先花 2 周验证 3 个核心假设。**

### 4.2 三个验证实验

#### 实验 1：NiceGUI 原型验证

| 项目 | 内容 |
|------|------|
| **目的** | 验证 NiceGUI 能否做出满意的仪表盘 |
| **任务** | 1 周内做出单页仪表盘原型 |
| **必须包含** | 标题区、KPI 卡片（4 个）、Agent 网格（3 个卡片）、日志面板（深色）、CEO 面板（输入框+发送按钮） |
| **验收标准** | 创始人打开原型，5 分钟内能说出"这就是我想要的"或"这里要改" |
| **通过** | 进入实验 2 |
| **不通过** | 切 Flutter，重新评估技术栈 |
| **Token 预算** | ¥50（本地为主，少量云端调用） |

#### 实验 2：Ollama 3B 基准测试

| 项目 | 内容 |
|------|------|
| **目的** | 验证 qwen2.5:3b 实际能覆盖多少场景 |
| **任务** | 用 20 个典型任务测试 |
| **测试集** | 意图解析（5 个）、SOP 选择（5 个）、代码生成（3 个）、内容创作（4 个）、数据分析（3 个） |
| **验收标准** | 成功率 ≥ 60%（即 12/20 个任务输出可用，无需人工重写） |
| **通过** | 按实际成功率调整成本模型，进入实验 3 |
| **不通过** | 如果 < 40%，考虑升级本地模型（7B）或增加云端预算 |
| **Token 预算** | ¥30（本地 0 Token，云端用于对比测试） |

#### 实验 3：内存压力测试

| 项目 | 内容 |
|------|------|
| **目的** | 验证 3 个 Agent + Ollama + ChromaDB 在 4GB 内存下是否流畅 |
| **任务** | 连续运行 1 小时典型工作负载 |
| **负载** | 主 Agent 解析意图 → 拉起开发 Agent 写代码 → 审计 Agent 检查 → Ollama 生成摘要 |
| **验收标准** | 内存峰值 < 3.5GB，无 OOM，UI 不卡顿 |
| **通过** | 冻结需求，进入设计阶段 |
| **不通过** | 减少常驻 Agent 到 2 个，或要求升级硬件（8GB 内存） |
| **Token 预算** | ¥20（本地为主） |

### 4.3 验证时间表

```
第 1 周
  ├── Day 1-2: 实验 1 —— NiceGUI 原型搭建
  ├── Day 3-4: 实验 1 —— 原型迭代，创始人验收
  └── Day 5: 实验 1 结论 → 通过/不通过

第 2 周
  ├── Day 1-3: 实验 2 —— Ollama 3B 20 任务基准测试
  ├── Day 4: 实验 2 结论 → 调整成本模型
  └── Day 5: 实验 3 —— 内存压力测试 → 最终结论

第 2 周末
  ├── 三个实验全部通过 → 更新 PRD 成本模型 → 冻结需求 → 进入 02-系统设计
  └── 任一实验不通过 → 调整方案 → 重新验证
```

### 4.4 验证预算

### 5.2 技术架构

| 层次 | 技术 | MVP 说明 |
|------|------|---------|
| 前端 | NiceGUI | 单页仪表盘，暗色主题唯一 |
| 后端 | Python 3.10+ | 创始人唯一熟悉的语言 |
| 结构化数据 | SQLite | 单文件，零配置 |
| 向量数据 | ChromaDB | 轻量使用，仅 Agent 记忆 |
| 本地模型 | Ollama | qwen2.5:3b，按需加载 |
| 云端模型 | Kimi/Coze/DeepSeek | 复杂任务备用 |
| 部署 | 本地桌面 | 数据不出门 |

### 5.3 Agent 架构

```
常驻 Agent（内存中）
  ├── 主 Agent —— 意图解析、SOP 选择、团队组建
  ├── 审计 Agent —— 代码检查、成本监督、提供建议（不裁决）
  └── 开发 Agent —— 代码生成、文件操作、工具调用

角色模板（磁盘上，需要时实例化）
  ├── 产品经理、架构师、测试 Agent
  ├── 营销 Agent、内容 Agent、运营 Agent
  └── 财务 Agent、行政 Agent、秘书 Agent

约束
  ├── 同时运行 Agent ≤ 3 个
  ├── Ollama 3B 按需加载，不用时卸载
  └── 总内存占用 ≤ 3GB（4GB 系统留 1GB 给 OS）
```
