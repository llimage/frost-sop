## 一、P0（阻塞）修复详情

### P0-1：F9 表结构字段不匹配 ✅

| 维度 | 详情 |
|------|------|
| **首审问题** | `energy_log`/`schedule`/`decision_points` 三张表的 CREATE TABLE 缺少部分列，依赖 migration 脚本补充 |
| **风险** | migration 脚本可能未执行，导致 INSERT/UPDATE 失败 |
| **修复方案** | 在 `core/db.py` 的 `_init_tables()` 中，将三张表的 CREATE TABLE 直接包含所有列 |
| **验证方式** | `test_f9_founder_tools.py`（13 tests）和 `test_f8_decision.py`（6 tests）全部通过 |
| **修复文件** | `core/db.py`（`energy_log`/`schedule`/`decision_points` 的 CREATE TABLE 语句） |

### P0-2：F8 决策管理逻辑回归 ✅

| 维度 | 详情 |
|------|------|
| **首审问题** | `resume_decision()` 引用不存在的 `tasks.decision_status` 列，UPDATE 语句执行失败 |
| **风险** | 决策恢复功能不可用 |
| **修复方案** | 移除对 `tasks.decision_status` 的引用；新增 `reject_decision()` 方法；添加 `threading.Lock` 线程安全保护 |
| **验证方式** | `test_f8_decision.py`（6 tests）全部通过 |
| **修复文件** | `core/decision_manager.py` |

### P0-3：真实 LLM 模式验证 ✅

| 维度 | 详情 |
|------|------|
| **首审问题** | 未验证真实 LLM 调用（仅 mock 模式测试） |
| **风险** | 生产环境可能无法正确调用 LLM |
| **验证结果** | 环境变量 `FROST_TESTING` 未设置 → 真实模式 → DeepSeek 成功响应 `"你好，我是你的测试助手..."`，prompt=20 + completion=20 = 40 tokens |

## 二、P1（严重）修复详情

### P1-5：架构定位文档 ✅

| 维度 | 详情 |
|------|------|
| **首审问题** | V2.0 定位不清——是"核心控制流事件驱动"还是"可观测性事件层"？ |
| **修复方案** | `V2_ARCHITECTURE.md` 顶部新增定位声明，明确 V2.0 是"可观测性事件层"，V3.0 将改造控制流 |
| **声明内容** | "V2.0 是「可观测性事件层」，不是「核心控制流事件驱动」。V3.0 将把控制流也改造为事件驱动。" |

### P1-6：TASK_DECOMPOSED 事件 ✅

| 维度 | 详情 |
|------|------|
| **首审问题** | 9 个事件类型中，`TASK_DECOMPOSED` 虽然定义了但从未被发布 |
| **修复方案** | `main.py` 中，SOP 内化完成后发布 `TASK_DECOMPOSED` 事件 |
| **事件数据** | `{"task_id": ..., "stage_count": N, "stages": [...]}` |
| **验证测试** | 3 个专项测试（发布验证/数据结构/阶段列表）全部通过 |

### P1-7：敏感数据过滤 ✅

| 维度 | 详情 |
|------|------|
| **首审问题** | event_log 持久化时没有过滤敏感数据（api_key/token/password 等） |
| **风险** | 敏感信息可能泄露到数据库中 |
| **修复方案** | `EventBus._sanitize_data()` 递归过滤 9 类敏感键（`api_key`, `apikey`, `token`, `password`, `secret`, `credential`, `private_key`, `access_key`, `auth`） |
| **过滤行为** | 匹配键的值替换为 `"[FILTERED]"`，递归处理嵌套 dict/list |
| **验证测试** | 7 个专项测试（基础过滤/嵌套/列表/非敏感保留/空数据边界）全部通过 |

### P1-8：循环事件防护 ✅

| 维度 | 详情 |
|------|------|
| **首审问题** | EventBus 没有防循环机制——如果某个 agent 的事件触发自身回调，可能导致无限循环 |
| **修复方案** | `publish()` 中跳过 `callback.__name__ == event.source` 的回调（排除 lambda，其 `__name__` 为 `<lambda>` 也跳过比较） |
| **验证测试** | 3 个专项测试（循环防护/lambda 不误杀/多回调场景）全部通过 |

### P1-9：agents 表 UPSERT ✅

| 维度 | 详情 |
|------|------|
| **首审问题** | `_write_agent_status()` 使用普通 INSERT，重复运行触发 UNIQUE 约束错误 |
| **风险** | E2E 执行日志中出现警告，虽然 fail-safe 但表示功能不完整 |
| **修复方案** | `_write_agent_status()` 改为 `INSERT OR REPLACE` |
| **验证测试** | 1 个专项测试（重复运行无 UNIQUE 约束错误）通过 |
