# V2.0 设计决策记录

> 记录 V2.0 改造过程中的关键架构决策及其理由

---

## 决策 1：为什么自建 EventBus 而不是引入外部依赖？

### 背景

事件总线是实现事件驱动架构的核心基础设施。市面上有多种成熟的 Python 消息队列/事件总线库。

### 考量因素

| 方案 | 优点 | 缺点 |
|------|------|------|
| 自建 EventBus | 零依赖、完全控制、定制化 | 需自行维护 |
| `blinker` | 成熟稳定、API 简洁 | 仅信号机制，无持久化 |
| `pypubsub` | 支持主题/消息 | 无线程安全保证、维护不活跃 |
| Redis Pub/Sub | 高可用、持久化 | 额外基础设施、运维复杂 |
| Kafka/RabbitMQ | 企业级可靠性 | 重型、过度设计 |

### 决策

**自建 EventBus**（`core/event_bus.py`, 322 行）

### 理由

1. **FROST-SOP 是轻量级单进程应用**，不需要分布式消息队列
2. **322 行代码量可控**，复杂度在可维护范围内
3. **定制化需求**：事件持久化（event_log 表）、EventType 常量类、fail-safe 语义——这些在通用库中难以实现
4. **减少外部依赖**：FROST-SOP 团队（1 人）维护外部依赖成本高
5. **线程安全**：Python `threading.Lock` 足够满足需求

## 决策 2：为什么使用同步分发而不是异步？

### 背景

EventBus.publish() 需要将事件分发给所有订阅者。可以选择同步（阻塞）或异步（非阻塞）分发。

### 考量因素

| 方案 | 优点 | 缺点 |
|------|------|------|
| 同步分发 | 简单、可预测、易调试 | 订阅者慢则阻塞整体 |
| 异步分发 | 不阻塞发布者 | 线程管理复杂、调试困难 |

### 决策

**同步分发**（`publish()` 中直接调用 handler）

### 理由

1. **FROST-SOP 任务执行本身就是同步的**，事件分发不应引入额外的异步复杂度
2. **订阅者逻辑简单**（主要是写 DB 和审计），不会长时间阻塞
3. **长老审计通过 `daemon=True` 后台线程实现**异步（不阻塞主流程），这是例外场景的专门处理
4. **调试友好**：同步调用链清晰，失败时栈追踪完整
5. **简单性原则**：不过度设计

## 决策 3：为什么采用渐进式改造（绞杀者模式）？

### 背景

V1.0 已在生产运行，不能因改造而中断服务。

### 考量因素

| 方案 | 优点 | 缺点 |
|------|------|------|
| 大爆炸重写 | 架构干净 | 高风险、长时间不可用 |
| 渐进式改造 | 低风险、随时可回滚 | 过渡期有冗余代码 |

### 决策

**绞杀者模式（Strangler Fig）**

### 理由

1. **`event_driven=False`（默认）保持 V1.0 行为**，用户无需任何迁移
2. **每个阶段独立提交、独立测试**，降低风险
3. **66 个新测试逐个加入，回归测试每次验证**，确保无退化
4. **未来可逐步移除 V1.0 相关代码**（当所有用户切换到 event_driven=True 后）

## 决策 4：为什么保留 event_driven=False 兼容模式？

### 背景

V2.0 引入了 `event_driven` 参数控制事件驱动行为的开关。

### 考量因素

| 方案 | 优点 | 缺点 |
|------|------|------|
| 始终启用事件驱动 | 简洁 | 破坏 V1.0 行为 |
| 可选启用的兼容模式 | 安全 | 增加一个参数 |

### 决策

**默认 `event_driven=False`，可选启用**

### 理由

1. **尊重现有用户**：不强制升级
2. **渐进式迁移**：用户可以逐个 Agent 开启事件驱动
3. **混合模式**：同一系统中可同时运行 V1.0 和 V2.0 模式的 Agent
4. **未来 V3.0 可改为默认 enabled**：当所有用户迁移完成后
5. **测试覆盖**：两种模式均有测试

## 决策 5：为什么长老审计通过后台线程而非直接在事件处理中执行？

### 背景

`subscribe_elder_to_events()` 注册的 handler 在 EventBus.publish() 中被同步调用。如果 handler 中直接执行 `audit_family()`（耗时操作），会阻塞事件分发。

### 考量因素

| 方案 | 优点 | 缺点 |
|------|------|------|
| handler 同步审计 | 简单 | 阻塞事件分发 |
| handler 启动后台线程 | 不阻塞 | 线程管理 |

### 决策

**handler 中同步调用，但内部通过 `finalize_task()` 启动 `daemon=True` 线程**

### 理由

1. **`daemon=True` 线程随主进程退出自动终止**，不泄漏
2. **审计失败不影响主流程**（fail-safe 设计）
3. **审计完成时的日志写入是安全的数据库操作**
4. **FROST-SOP 是单用户系统**，不存在并发审计冲突

## 决策 6：为什么事件持久化使用 SQLite event_log 表？

### 背景

EventBus 需要持久化事件，供事后审计和分析。

### 考量因素

| 方案 | 优点 | 缺点 |
|------|------|------|
| SQLite event_log 表 | 一致性好、查询方便 | 写入性能较低 |
| 文件日志 | 简单 | 难以查询和统计 |
| Redis | 高性能 | 额外基础设施 |

### 决策

**SQLite event_log 表**

### 理由

1. **与 FROST-SOP 现有数据存储一致**（SQLite）
2. **支持 SQL 查询**：按事件类型、时间范围、来源等查询
3. **事务一致性**：事件持久化和任务数据在同一数据库中
4. **event_log 增长速度可控**（每次任务约 20-50 条事件）
5. **可定期归档**：建议保留最近 30 天

## 决策 7：为什么 EventBus 使用 Double-Checked Locking 实现单例？

### 背景

EventBus 需要全局唯一实例（单例），且需线程安全。

### 考量因素

| 方案 | 优点 | 缺点 |
|------|------|------|
| 简单锁 | 安全 | 每次获取都加锁，性能损耗 |
| Double-Checked Locking | 安全 + 性能 | 略微复杂 |
| 模块级变量 | 最简单 | 非线程安全 |

### 决策

**Double-Checked Locking**

```python
with _lock:
    if _instance is None:
        _instance = EventBus()
```

### 理由

1. **首次检查（无锁）**：绝大多数情况下实例已存在，无需加锁
2. **二次检查（加锁）**：确保只有一个线程创建实例
3. **Python GIL + threading.Lock**：在此场景下安全

## 决策总结

| # | 决策 | 选择 | 驱动因素 |
|---|------|------|----------|
| 1 | 事件总线实现 | 自建 EventBus | 轻量级、可控、零依赖 |
| 2 | 分发方式 | 同步分发 | 简单、可预测、FROST-SOP 同步特性 |
| 3 | 改造策略 | 绞杀者模式 | 低风险、随时可回滚 |
| 4 | 兼容模式 | event_driven=False 默认 | 尊重现有用户 |
| 5 | 审计触发 | 后台线程 | 不阻塞主流程 |
| 6 | 事件持久化 | SQLite event_log 表 | 一致性好、可查询 |
| 7 | 单例安全 | Double-Checked Locking | 性能 + 线程安全 |
