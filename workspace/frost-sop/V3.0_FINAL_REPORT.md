## 2. P1-1 修复状态

**问题**：同步 `Agent.run()` 在异步回调中直接调用，阻塞事件循环（真实 LLM 模式下）。

**修复方案**：将 `Agent.run()` 包装为 `await asyncio.to_thread()`。

| 文件 | 位置 | 修复方式 | 状态 |
|------|------|----------|------|
| `agents/ancestor.py` | `on_task_created` (L77) | `await asyncio.to_thread(lambda: ancestor.run(...))` | ✅ |
| `agents/parent.py` | `on_task_decomposed` (L127, L158) | `await asyncio.to_thread(functools.partial(parent.run, ...))` | ✅ |
| `skills/orchestration.py` | `register_stage_executor` (L544) | `await asyncio.to_thread(execute_stage, stage_context)` | ✅ |

**验证**：
- V3.0 测试 36/36 通过（mock 模式）
- P1-1 专项测试 `test_p1_1_fix_ancestor_run_not_blocking` 通过
- 事件循环阻塞测试 `test_no_blocking_during_async_call` 通过

## 5. 已知限制（同 V3_ACCEPTANCE_REPORT.md）

1. **`event_driven=True` 为 opt-in** — 需传 `--async-mode` 参数启用，默认 V2.0 管道模式
2. **Streamlit UI 保持 V2.0 管道模型** — V3.0 事件驱动未接入 UI（计划 V3.1）
3. **中断恢复延后到 V3.1** — 当前无任务中断后的状态恢复机制
4. **订阅者累积问题** — `main_async()` 多次运行后订阅者数量累积（Known Issue）
5. **`publish()` 顺序执行** — 未利用 `asyncio.gather()` 并发执行订阅者回调
