## 阶段1.2: 人类府兵超时机制 ✅

### 问题
- 人不一定会及时跟进
- 和人之间的进度交互以及整个 SOP 的超时容忍度是一个问题

### 修复
1. **数据库迁移** (`core/db.py`):
   - `timeout_minutes`: INTEGER DEFAULT 30
   - `expires_at`: TIMESTAMP
   - `timeout_action`: TEXT DEFAULT 'escalate'

2. **超时检查函数** (`core/db.py`):
   - `check_expired_decisions()`: 查找已超时的 pending 决策点
   - `resolve_decision_timeout()`: 处理超时决策（escalate/abort/auto_approve）

3. **API 端点** (`api/main.py`):
   - `POST /api/decisions/check-timeout`: 检查并处理超时决策
   - 返回: `{ checked, resolved, failed, timestamp }`

### 审计
- 7/7 检查项通过

## 阶段1.3: API 错误处理标准化 ✅

### 问题
- 多个空 except 块导致静默失败
- 没有全局异常处理

### 修复
1. **全局异常处理器** (`api/main.py`):
   - `@app.exception_handler(Exception)`
   - 生成唯一 error_id
   - 结构化错误响应: `{ error_code, error_id, message, path }`
   - HTTPException 保留原始状态码

2. **空 except 块清理** (6个):
   - 第438行: 预算配置读取 → 降级策略 + WARN 日志
   - 第551行: SSE audit_log 查询 → WARN 日志 + 继续重试
   - 第796行: SSE tasks 查询 → WARN 日志
   - 第816行: SSE stages 查询 → WARN 日志
   - 第830行: SSE decisions 查询 → WARN 日志
   - 第844行: SSE agents 查询 → WARN 日志
   - 第925行: SOP 模板加载 → 降级 + WARN 日志

3. **get_task_stages 异常处理**:
   - 输入校验: 空 task_id → 400 TASK_ID_INVALID
   - DB 查询失败 → 500 STAGES_QUERY_FAILED + 结构化日志
   - 资源限制: LIMIT 100

### 审计
- 0 个空 except 块
- 10 个错误码定义
- 2 个 ERROR 日志标记
- 7 个 WARN 日志标记

## 阶段1.4: 前端字数限制修复 ✅

### 问题
- 对话输入有字数限制，长需求被截断
- 单行 Input 不方便输入长文本

### 修复
1. **前端 CeoChat.tsx**:
   - `<Input>` → `<Textarea>` (多行文本域)
   - `Shift+Enter` 换行, `Enter` 发送
   - 实时字数统计: `{input.length} 字`
   - 可调整大小: `resize-y`, `max-h-[200px]`

2. **后端 chat 端点** (`api/main.py`):
   - `_max_tokens`: 1024 → 4096

### 审计
- 8/8 前端检查通过
- 2/2 后端检查通过
