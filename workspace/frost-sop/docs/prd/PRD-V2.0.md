## 2. 产品概述

### 2.1 产品定位

**FROST-SOP** 是一个专为 **一人公司（One-Person Company, OPC）** 设计的 AI 指挥平台。它不是一个代码生成工具，也不是一个直接操作外部平台的工具——它是一个**编排系统**，通过 **Store + Skill + SOP** 三大核心机制，将 AI Agent 组织成可协作的"家族"，在标准化流程中自动执行复杂任务，同时把人类创始人（你）放在指挥位置。

**核心哲学**: AI 是杠杆而非替身。人始终是君主，AI 是府兵。

### 2.2 产品目标（三个层次）

| 层次 | 目标 | 说明 |
|------|------|------|
| **验证层** | 验证 frost 哲学 | 证明 Store+Skill+SOP 的架构可以支撑真实业务运转 |
| **自用层** | 作为廖亮的 OPC 运行平台 | 覆盖心域探险、长护险培训、心栖陪伴、心理歌曲等项目的全过程管理 |
| **产品层** | 作为 OPC 运行平台产品销售 | 让其他 OPC 创始人也能使用，适应不同业务需求 |

### 2.3 目标用户画像

**主要用户**: 一人公司创始人（OPC Founder）
- 技术背景: 非专业开发者（自称"IT小白"），但愿意使用 API Key 和自动化脚本
- 工作方式: 追求秩序感，倾向通过内容产品建立间接影响力，排斥 1V1 服务
- 资源约束: 月度 Token 预算 ¥300，需要严格的成本控制
- 核心诉求: 高自动化、低维护、数据主权、隐私保护

**扩展用户**: 小型团队（2-5人）
- 多个 OPC 创始人协作
- 需要多项目之间的资源共享和协同

### 2.4 产品边界（明确不做什么）

| 不做 | 原因 | 替代方案 |
|------|------|----------|
| **不写代码** | 平台是编排代码的，但不生成可部署的代码 | 通过 Skill 调用外部代码生成工具 |
| **不直接操作外部平台** | 发布到掘金/邮件等通过 Skill 间接做 | 外部平台联系通过 Skill + SOP 搞定，不硬编码 |
| **不处理支付/收款** | 支付是业务内部机制，不在平台层面 | 业务需要时自行集成 |
| **不做家族间指挥机制** | 多家族协同是资源共享，不是层级指挥 | 多项目共享 Store 和 Skill |

## 3. 需求分析

### 3.1 功能性需求

#### 3.1.1 核心机制: Store + Skill + SOP

**Store（资产库）**
- 存储所有持久化数据: 任务记录、执行结果、教训归档、SOP 模板、Skill 定义
- 支持版本管理: SOP v1 → v2 保留回滚能力
- 支持多项目隔离 + 跨项目共享

**Skill（技能）**
- 可复用的原子能力: LLM 调用、数据抓取、文件处理、通知发送
- 触发机制: 关键词匹配、SOP 阶段调用、API 直接调用
- 版本进化: 基于执行反馈自动优化（自进化）
- 外部 Skill 引用: 可以引用或修改社区/外部 Skill

**SOP（标准操作程序）**
- 定义任务执行的流程: 阶段序列、Agent 分配、超时设置、人类决策点
- 阶段类型: 自动执行、等待人类决策、等待外部输入、条件分支
- 模板化: 可复用的 SOP 模板库
- 执行追踪: 每个阶段的输入、输出、耗时、成本

#### 3.1.2 家族 Agent 体系

基于 frost 哲学的 Agent 组织方式:

```
创始人（君主）
    ↓ 发布任务
Ancestor（祖辈）— 编排 SOP、分配任务
    ↓ 派遣
Parent（父辈）— 执行协调、监控进度
    ↓ 派遣
Scout（斥候）— 具体执行、收集信息
    ↓ 回报
Hunter（猎手）— 结果处理、知识提取
```

**Agent 代际限制**:
- 防止无限递归: Agent 不能无限 spawn 子 Agent
- 合并机制: 子 Agent 完成后可以选择性持久化或合并回父 Agent

#### 3.1.3 人类府兵机制

**核心概念**: 在 SOP 中，需要人类介入的时候，把人当作一个"府兵"来看待——人有自己的状态（idle/busy）、有超时限制、有任务队列。

**人类作为府兵的特征**:
- 状态: `idle`（空闲）/`busy`（忙碌）/`offline`（离线）
- 超时: 默认 30 分钟，可配置
- 超时动作: `escalate`（升级）/`abort`（中止）/`auto_approve`（自动批准）
- 进度交互: 系统主动推送待办，人类通过对话界面响应
- 任务队列: 多个等待决策按优先级排序

**人类作为君主的特征**:
- 发布任务: 通过自然语言描述愿景，系统初始化 SOP
- 监控全局: 查看所有家族状态、项目进度、成本消耗
- 战略决策: 批准/驳回系统提出的优化建议（自进化）

#### 3.1.4 愿景初始化与防飘逸

**问题**: LLM 对同一愿景的理解可能每次不同，导致执行结果不一致。

**解决方案: 多轮问卷澄清**

```
用户输入: "创建一个AI在线培训项目"

系统响应:
"收到愿景。为了精准执行，请澄清以下问题:
1. 培训的目标受众是谁？（初学者/进阶者/专家）
2. 培训形式是什么？（视频/文字/互动/混合）
3. 预计课程数量？
4. 有无特定技术栈要求？
5. 交付时间期望？"

用户回答后 → 系统生成结构化愿景文档 → 用户确认 → 初始化 SOP
```

**愿景文档结构**:
```json
{
  "vision_id": "vis_xxx",
  "raw_input": "原始输入",
  "clarified_questions": [
    {"question": "...", "answer": "..."}
  ],
  "structured_goal": "结构化目标",
  "success_criteria": ["标准1", "标准2"],
  "constraints": ["约束1", "约束2"],
  "suggested_sop": "SOP-XXX",
  "confidence_score": 0.92
}
```

#### 3.1.5 自进化能力

**核心流程**: 错误记录 → 教训归档 → 复盘分析 → 系统进化

**阶段1: 错误检测与记录**
- 自动捕获: 所有异常、失败、超时自动记录
- 结构化: 错误类型、组件、上下文、堆栈
- 去重: 相同错误合并，增加计数

**阶段2: 教训归档**
- LLM 生成结构化教训: severity, category, root_cause, actionable_fix
- 存储到 lesson: 前缀的 Store key
- 关联到 plan/phase/skill

**阶段3: 趋势分析**
- 统计: 成功率、失败率、主要错误类型
- 洞察: 哪个 SOP 失败率高、哪个 Skill 不稳定
- 生成优化建议

**阶段4: 复盘决策（人类确认）**
- 生成报告: 数据概览 + 趋势洞察 + 优化建议
- 创始人决策: 批准/驳回/修改每条建议
- 超时: 30 分钟无响应自动 escalate

**阶段5: 系统进化**
- SOP 优化: 创建 v2 版本，保留 v1 回滚
- Skill 更新: 基于失败案例调整 prompt/参数
- 配置调整: 超时时间、重试策略、预算阈值

**阶段6: 验证与回滚**
- 自动化测试验证进化后的系统
- 失败则自动回滚到上一版本

#### 3.1.6 隐私保护与本地 LLM

**策略**: 本地轻量化 LLM + 后端 LLM 脱敏

**本地 LLM 职责**:
- 愿景澄清: 多轮对话，不需要高智能
- 简单分类: 关键词匹配、意图识别
- 数据预处理: 脱敏、摘要、格式化

**后端 LLM 职责**:
- 复杂推理: SOP 编排、代码审查、内容生成
- 接收的是脱敏后的数据

**本地 LLM 选型**（基于调研）:
- Ollama: 轻量级框架，支持 Qwen/DeepSeek/Llama 系列
- 推荐模型: Qwen3-1.7B/3B（中文好、轻量化）、DeepSeek-R1-1.5B（推理）
- 硬件要求: 最低 8GB 内存，推荐 16GB
- 量化: 4-bit 量化减少显存占用

**脱敏策略**:
- 个人信息: 姓名 → [NAME], 电话 → [PHONE]
- 业务敏感: 具体金额 → [AMOUNT], 客户名 → [CLIENT]
- 保留语义: 脱敏后仍能用于分析和生成

#### 3.1.7 成本控制

**预算体系**:
- 月度预算: 默认 ¥300（可配置）
- 熔断机制: 单日消耗超过 ¥100 自动暂停非关键任务
- 成本追踪: 每个任务、每个阶段、每个 Agent 的 Token 消耗和费用
- 预警: 预算使用 50%/80%/100% 时通知

**成本优化策略**:
- 模型选择: 简单任务用本地 LLM，复杂任务用云端 LLM
- 缓存: 相同输入的 LLM 调用结果缓存
- 批量: 多个小任务合并为一次 LLM 调用
- 降级: 预算紧张时自动降低模型规格

#### 3.1.8 多家族协同（多项目共享）

**概念**: 不同项目 = 不同家族，共享 Store 和 Skill，但不共享任务执行状态。

**共享内容**:
- Skill 库: 所有家族共用，一个 Skill 改进所有家族受益
- SOP 模板: 通用模板跨项目复用
- 教训库: 一个项目的失败教训可以预警其他项目
- 成本池: 统一预算管理

**隔离内容**:
- 任务执行状态: 各家族独立
- Agent 实例: 各家族独立生命周期
- 项目数据: 各家族独立存储

**协同场景**:
- 项目 A 的"内容生成"Skill 优化后，项目 B 自动使用新版本
- 项目 A 的"合规审查"失败教训，项目 B 执行时自动提示
- 创始人可以同时查看多个家族的仪表盘

### 3.2 非功能性需求

#### 3.2.1 可用性

- **前端必须好用**: 这是产品的核心痛点，必须有一个流畅、直观、响应迅速的前端
- **移动端适配**: 创始人可能通过手机查看进度、做决策
- **暗色模式**: 长时间使用的舒适性
- **实时反馈**: SSE 实时推送状态变更，无需手动刷新

#### 3.2.2 性能

- **启动时间**: 前端首屏 < 3s，后端 API 响应 < 500ms
- **并发**: 支持 5 个家族同时运行，每个家族 10 个并行任务
- **SSE 延迟**: 状态变更到前端显示 < 2s
- **LLM 调用**: 超时 30s，重试 3 次，指数退避

#### 3.2.3 可靠性

- **零静默失败**: 所有异常必须有日志、错误码、处理策略
- **自动恢复**: SSE 断线自动重连（最多 10 次，指数退避）
- **数据持久化**: SQLite + 定期备份，防止数据丢失
- **回滚能力**: SOP v1 → v2 失败可回滚到 v1

#### 3.2.4 安全性

- **输入校验**: 所有外部输入必须校验，防止注入
- **数据隔离**: 多项目数据严格隔离
- **API 安全**: CORS 限制、请求频率限制
- **隐私**: 本地 LLM 处理敏感数据，后端 LLM 只接收脱敏数据

#### 3.2.5 可维护性

- **自进化**: 系统能从自己的错误中学习，减少人工维护
- **模块化**: Store/Skill/SOP 独立演进，不影响其他模块
- **文档化**: 每个 Skill、每个 SOP 都有文档和示例
- **测试覆盖**: 单元测试 + 集成测试 + 回归测试

## 4. 外部调研总结

### 4.1 AI Agent 编排平台分析

| 平台 | 类型 | 优点 | 缺点 | 与 FROST-SOP 的关系 |
|------|------|------|------|---------------------|
| **CrewAI** | 多 Agent 协作 | 角色定义清晰、易用 | 缺乏持久化、无自进化 | 参考角色模型，但需自建持久化 |
| **AutoGen** | 对话式 Agent | 微软背书、功能丰富 | 复杂、学习曲线陡 | 参考对话机制，但简化使用 |
| **LangGraph** | 图编排 | 可视化、灵活 | 需要代码能力 | 参考图结构，但 SOP 更简化 |
| **n8n** | 工作流自动化 | 自托管、400+集成 | 非 AI 原生、开发者向 | 参考工作流概念，但 AI 优先 |
| **FlowSteer** | RL 进化 | 端到端学习 | 研究阶段、不稳定 | 参考自进化思想，但规则+LLM 混合 |
| **GraphBit** | 图 Agent | 确定性编排、避免 LLM 飘逸 | 新兴、生态小 | 参考确定性编排，避免纯 LLM 决策 |

### 4.2 本地 LLM 部署方案

| 方案 | 适用场景 | 硬件要求 | 与 FROST-SOP 的关系 |
|------|----------|----------|---------------------|
| **Ollama** | 个人本地运行 | 8GB+ 内存 | **推荐**: 简单、跨平台、OpenAI 兼容 API |
| **LM Studio** | GUI 用户 | 8GB+ 内存 | 备选: 图形界面友好 |
| **llama.cpp** | 极致性能 | 因模型而异 | 底层: 其他方案的底层实现 |
| **vLLM** | 高并发服务 | GPU 推荐 | 后端: 如果本地需要高并发 |

### 4.3 关键洞察

1. **确定性编排优于纯 LLM 编排**: GraphBit 的研究表明，将 LLM 限制在领域特定推理，由确定性引擎处理编排，可以避免系统失败。FROST-SOP 的 SOP 引擎应该是确定性的，LLM 只负责阶段内的推理。

2. **Skill 进化是差异化**: FlowSteer 的 RL 方法虽然先进但不稳定。FROST-SOP 采用规则+LLM 混合: 失败模式由规则匹配，优化建议由 LLM 生成，人类确认后执行。

3. **本地 LLM 是隐私刚需**: 所有调研都强调本地部署的隐私优势。Ollama 是最适合非技术用户的方案，与 FROST-SOP 的"IT 小白"用户画像匹配。

4. **工作流平台缺乏 AI 原生**: n8n/Zapier 等传统工作流平台不是为 AI Agent 设计的。FROST-SOP 的机会在于"AI 优先的编排"。

## 5. 风险与假设

### 5.1 风险

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| LLM 成本超预算 | 中 | 高 | 熔断机制、本地 LLM 降级、缓存 |
| 前端开发瓶颈 | 高 | 高 | 使用现成组件库、优先核心功能 |
| 本地 LLM 性能不足 | 中 | 中 | 量化模型、任务分级、云端降级 |
| 自进化效果不佳 | 中 | 中 | 人类确认环节、回滚机制、渐进式 |
| 多家族协同复杂 | 低 | 高 | 先单家族稳定，再扩展 |

### 5.2 假设

- 用户有基本的 API Key 使用经验（能购买和配置）
- 用户硬件至少 8GB 内存（运行本地 LLM）
- 用户愿意花 1-2 小时完成初始配置
- 后端 LLM 服务（DeepSeek/OpenAI）可用且稳定

## 6. 验收标准

### 6.1 功能验收

- [ ] 创始人输入愿景，系统通过多轮问卷澄清，生成结构化愿景文档

### 6.2 非功能验收

- [ ] 前端首屏 < 3s，API 响应 < 500ms
- [ ] SSE 状态同步延迟 < 2s
- [ ] 零静默失败: 所有异常有日志、错误码、处理策略

- [ ] 本地 LLM 在 8GB 内存机器上可运行
